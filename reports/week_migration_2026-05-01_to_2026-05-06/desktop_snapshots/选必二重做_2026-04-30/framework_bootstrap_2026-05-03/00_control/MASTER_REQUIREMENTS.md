# 选必二《法律与生活》框架生成总要求

日期：2026-05-03

## 任务目标

从新选必二线的预处理原料层出发，边复核 2024-2026 北京区卷法律题，边生成并验证一个可教学、可迁移、可容纳新题的《法律与生活》框架。

最终目标不是把教材目录重写一遍，而是形成和用户前几本书一致的考试框架：

材料触发信号 -> 法律关系与争点 -> 法定要件/规则 -> 责任/效力/程序 -> 答案落点与价值收束。

## 最高边界

- 旧选必二线全部作废，不继承旧结论、旧框架、旧题型、旧清单或旧错肢。
- 可读取旧文件的唯一例外：用户明确要求做历史对比或追责；本轮不需要。
- 本轮以当前新线预处理成果和原始 2024/2025/2026 模拟题源为依据。
- 普通参考答案不能升格为评分细则；主观题框架节点优先来自评分细则、评标、阅卷总结、讲评给分口径。
- 选择题只有在题面和可靠客观答案键均锁定后，才能写错项规律。

## 框架验收标准

框架必须同时满足两个要求：

- 主干：优先呈现考频最高、最常进入评分细则、最能迁移到新情境的法律思维和答题动作。
- 全面：所有已锁定选必二题目必须能归位；新题或新细则归不进去时，必须记录为框架缺口并触发修订。
- 命题路径：框架必须能够解释题目是如何被命制出来的，即命题人如何从法治思维培养目标出发，选择案例/法条/法律现象，设计设问路径，并把评分细则落到具体答题动作。

本轮吸收的核心命题启发：

- 法律模块命题要在法律学理、知识、政治价值、建设批判、理论实践之间取得统一。
- 选择题常以案例/法条为载体，考制度本身内容和细节；不能只滑向宏观价值。
- 主观题常以案例/法条/法律现象为载体，考法律知识如何推动法治建设与社会经济发展；不能只做复杂案件分析，也不能混同为必修三宏观法治题。
- 因此每道题必须新增 `proposition_path` 分析，先看命制逻辑，再做框架归位。

## 候选框架 v0 脚手架

一核：以事实为根据、以法律为准绳，定分止争。

二线：
- 权利保护与权利边界。
- 法治规则与德治价值。

三问：
- 判什么/怎么处理。
- 凭什么。
- 有什么意义。

四步：
- 定主体关系。
- 找争点事实。
- 套法定要件。
- 落责任/效力/程序。

五域：
- 合同、消费者、劳动。
- 物权、相邻、继承、家庭。
- 人格权、侵权。
- 知识产权、不正当竞争。
- 纠纷解决、生态公益、新业态。

该 v0 只是脚手架。任何节点都可以被题源和细则推翻、拆分、合并或改名。

## 执行节奏

每一批处理若干套试卷或一个题型簇，必须产出：

- 题目归位矩阵。
- 主干考频统计。
- 框架覆盖率和缺口清单。
- Codex A 证据结论。
- ClaudeCode B 独立结论。
- GPT-5.5 Pro 框架压力测试记录。
- Claude Opus 4.7 教学结构/迁移性建议。
- Codex 本地裁决。

## 框架自我进化机制

本轮框架不是一次性制定，而是一个批次驱动的自我进化系统。每次框架升级必须经过 `Framework Council`：

1. `Evidence Pack`：Codex 从当前批次生成证据包，包括题目案例卡、命题路径反推、细则摘要、频次统计、当前框架归位矩阵、归位困难题、框架缺口。
2. `GPT Independent Blueprint`：真实调用 GPT-5.5 Pro，让其独立研究“主干高频、全面覆盖、风险节点、版本升级条件”。
3. `Claude Independent Blueprint`：真实调用 Claude Opus 4.7 Adaptive，让其独立研究“学生可学性、框架层级、可背可迁移表达、过抽象风险”。
4. `Cross-Critique`：把 GPT 原始方案发给 Claude Opus 反驳/吸收；把 Claude 原始方案发给 GPT 反驳/吸收。两者必须回应对方，不接受孤立点评。
5. `Convergence Brief`：Codex 汇总双方一致点、冲突点、建议新增/删除/改名节点、需要本地证据核验的项目。
6. `Local Evidence Decision`：Codex 回到题面和细则裁决。只有能改善“主干清晰度、全题覆盖、困难题归位、学生迁移、证据边界”的建议才能进入新版本。
7. `Version Bump`：每次采纳修改都生成新框架版本，并同步更新覆盖矩阵、缺口清单和模型建议日志。

框架版本不得只靠 Codex、GPT 或 Claude 任一方单独升级。真实 GPT 和真实 Claude 未完成 cross-critique 时，最多标为 `candidate_pending_real_call`。

框架必须始终保留两层：

- `主干高频层`：高频、评分细则反复出现、学生必须优先掌握的答题动作。
- `开放容器层`：低频、边界、新题、新细则可暂时归位的容器；若后续频次上升，再升级为主干。

## 模型权限

- Codex：本地总控、证据裁决、生产线 A、融合、输出。
- ClaudeCode：生产线 B，独立复核题源、归位、查漏，不得只做审稿。
- GPT-5.5 Pro：战略审框架、压力测试、缺口识别、内容审稿。不是证据权威。
- Claude Opus 4.7 Adaptive：框架表达、教学路径、迁移性、学生版语言。不是证据权威。

GPT-5.5 Pro 和 Claude Opus 4.7 Adaptive 必须真实调用。不得用 Codex 子智能体、本地模拟角色、普通 advisor 文件或 Codex 自写“仿 GPT/Claude 意见”替代。Codex 子智能体只能准备上下文包、问题清单和本地预检查，不能计入 GPT/Claude 审议。

GPT 或 Claude 的任何建议必须进入 `MODEL_ADVICE_LOG`，由 Codex 回到本地题源和细则验证后才能采纳。若真实调用未完成，本阶段状态只能标为 `real_call_pending` 或 `blocked_advisor`，不能宣布框架通过、内容审稿通过、Governor PASS 或最终闭环。

## 会话隔离

- 本项目不得复用正在服务其他项目的 GPT-5.5 Pro、Claude Opus 4.7 Adaptive 或 ClaudeCode 对话/线程/工作目录。
- 每次真实模型调用都必须新开独立会话，提示词首部标明：`项目：选必二《法律与生活》框架实验；禁止继承其他项目上下文`。
- ClaudeCode 若参与本轮，必须使用本项目专用 worktree/目录和专用输出目录，不得接入其他项目的生产线。
- 若独立性无法确认，相关模型 gate 标记为 `advisor_session_isolation_uncertain`，重开会话后再计入真实调用。
- 用户要求的正式模型 gate 是 user-visible `GPT-5.5 Pro` 与 `Claude Opus 4.7 Adaptive Thinking`。CLI/API 非交互输出只能记为 `cli_provisional_advice`，不能替代网页端或桌面端可见会话；未完成时标记 `web_visible_pro_adaptive_call_pending`。

## 完成定义

不能只说“框架定了”。完成必须同时满足：

- 所有已锁定主观题都有框架归位。
- 所有答案已锁定的选择题都有错项规律归位。
- uncertain / 缺 OCR / 未锁答案项目仍在缺口清单中，没有被冒充闭环。
- 框架主干有频次和来源题支撑。
- 每个框架节点都有至少一个真实题源样本；没有样本的节点只能列为候选或未来容器。
- GPT-5.5 Pro 和 Claude Opus 的分歧已记录，Codex 已作本地裁决。
- Governor 通过“主干 + 全面 + 证据边界”检查。
