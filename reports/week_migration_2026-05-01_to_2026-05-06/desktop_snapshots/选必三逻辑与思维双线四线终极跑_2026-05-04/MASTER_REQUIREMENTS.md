# Master Requirements

## Highest-Priority User Requirements

- 本轮对象是选择性必修三《逻辑与思维》，分成两条成果线：
  - `思维部分`：按必修四哲学最终工作流重跑。思维和方法要像哲学“原理和方法论”一样处理，核心是材料触发、为什么想到、答案落点、学生迁移。
  - `推理部分`：学习用户上传框架，按题型族群总结常考考题和解决办法，同类题放一起，例如三段论、假言推理、选言推理、归纳推理、类比推理等。
- 用户上传的两份 PDF 是本轮框架源：
  - `逻辑与思维_思维部分_原文件.pdf`
  - `逻辑与思维_推理部分_原文件.pdf`
- 思维部分最终稿标准完全对齐先前哲学宝典最终工作流：四线协作、逐题证据闭环、学生版/审计版分离、最终 Markdown + Word/PDF 交付、Governor、Confucius、GPT-5.5 Pro 内容审稿。
- 必须使用 Codex、ClaudeCode、Claude Opus、GPT-5.5 Pro 四线：
  - Codex 自己也跑，不能只当总控。
  - ClaudeCode 是独立生产 lane，不是只做审阅。
  - Claude Opus 负责证据锁定后的教学化表达。
  - GPT-5.5 Pro 负责具体内容审查和压力测试。
- 外部 GPT/Claude 调用必须新开或避开正在占用的旧对话，不能把旧项目上下文混入本轮。
- 旧选必三稿只可作为定位索引和历史问题提醒；新增/保留的题链必须回到本地题源、答案、细则、评标、讲评或可靠客观答案源核验。
- 2026 年材料必须按 2026-05-04 当前本机 source roots 重新扫描，不继承旧稿“2026二模未开考”等时间判断。

## Evidence And Boundary Rules

- 主观题以正式细则、评分参考、评标、阅卷细则、阅卷总结、用户确认可用的评分材料为最高证据。
- 讲评 PPT 若有明确给分口径，只能标为 `A-support`。
- 普通参考答案、教师版答案、题面信号、分类汇编、讲评截图不能冒充正式细则。
- 选择题必须同时有题面、四个选项和可靠客观答案源；不能用旧稿或解释反推答案。
- 纯形式逻辑和推理规则题进入`推理部分题型稿`，不塞进`思维部分主链`。
- 如果同一题前半是推理、后半是思维方法应用，分开处理。
- `2026石景山期末`继续排除，除非用户提供新的可用评分细则。

## Student-Facing Contract

- 思维部分条目按学生思考顺序写：`材料动作/触发点 -> 完整设问 -> 为什么想到 -> 思维方法/方法论 -> 答案句`。
- 每个思维方法都要落到用户框架节点，不能只是概念讲义。
- 推理部分按题型族群写：`识别入口 -> 规则 -> 常见设错 -> 解题步骤 -> 例题变式 -> 避坑口令`。
- 学生版不得出现 source path、line id、OCR/debug、A-formal、pass、filled、评标、参考答案、可从……角度作答等审计话术。
- 选择题如进入学生版，应显示四个选项并分行排版。
- Word 交付必须渲染检查；如果 Microsoft Word 可用，需要打开/保存/验收。

## Final Gates

- `outline`、`section_batch`、`final_markdown`、`word_pdf` 都要经过 GPT-5.5 Pro 内容审稿或有明确 fallback/waiver。
- Claude Opus 教学化稿件不能引入未经本地证据支持的新知识点。
- Governor 必须检查证据边界、模块边界、学生版洁净度、图片/表格/推理题型覆盖、Word 版式。
- Confucius 必须只读最终学生稿，验证零基础学生是否能迁移到新题。
- `FINAL_ACCEPTANCE_REPORT.md` 只有在所有门都通过时才能写 `TASK_COMPLETE`。
