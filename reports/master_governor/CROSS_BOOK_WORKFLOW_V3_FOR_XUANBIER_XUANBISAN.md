# Cross-Book Workflow V3 For Xuanbier And Xuanbisan

Date: 2026-05-18

This workflow captures the process fixes learned from the xuanbiyi handbook rebuild. It applies before any future full-book or handbook-style production for:

- 选必二《法律与生活》
- 选必三《逻辑与思维》

It is a workflow and evidence-control rule. It does not replace each book's hard-rule notebook, source hierarchy, or module-specific skill.

## 0. Mandatory Entry Gate

Before touching source files, rubrics, generated text, Word output, or sync artifacts, the worker must pass all three gates:

1. Project gate: read `reports/master_governor/latest_master_governor_report.md`, `worker_daily_orders.md`, `PROJECT_GOVERNOR_THREE_LAYER_SOP.md`, and this file.
2. Book gate: read the active book skill and its hard-rule notebook.
3. Run gate: read or create the current run's `TASK_BRIEF.md`, `DEVELOPMENT_PLAN.md`, `PROGRESS.md`, source ledger, coverage matrix, and governor/acceptance files.

If the run is missing from the master governor report, stale, missing coverage, missing governor, or flagged possible false closure, repair the control layer before content production.

## 1. Source Packet First

Do not write a handbook entry until a source packet exists.

Every candidate question must first be locked with:

- paper and answer/rubric source file paths;
- full question prompt;
- material text or page-image pointer;
- scoring/rubric/marking/lecture source excerpt;
- evidence level;
- module-boundary decision;
- missing/OCR/conversion blockers if any.

Reference answers are support only unless the user confirms they are scoring material. Do not promote ordinary reference answers into formal rubric terms.

## 2. Evidence Ledger

Every model call, source decision, and final claim must leave a durable record.

Minimum fields:

- step id;
- batch/question ids;
- input packet path;
- output path;
- model/tool identity;
- URL or CLI command;
- raw response/log path;
- adjudication result;
- accepted/rejected/deferred reason;
- Git commit if the result is synced.

If the evidence is not saved, the final report must not claim that step happened.

## 3. Model Authority Chain

For handbook-style final production, use this order unless the user explicitly changes it:

1. Codex creates an independent thick draft from the source packet.
2. ClaudeCode creates an independent thick draft from the same source packet, without reading Codex's conclusions.
3. Codex builds a diff packet: terms, framework placement, trigger logic, answer sentence, source confidence, and missing cases.
4. GPT Pro is the primary fusion editor. It must receive both thick drafts plus the diff packet and output a usable fused draft, not merely a review memo.
5. Codex performs evidence backcheck only: remove unsupported additions, format to the target structure, and write adjudication logs.
6. Claude Opus 4.7 Adaptive may run after GPT Pro fusion for teaching language, misclassification risk, and student-transfer risks. It does not replace GPT Pro's primary fusion role unless the user says so.
7. Governor accepts only when source fidelity, model evidence, module boundaries, and final file rendering all pass.

If any lane is unavailable, mark the phase `blocked_advisor`, `real_call_pending`, or `user_waived_real_call`. Do not silently downgrade to a simulated advisor.

## 4. ClaudeCode Hard Lock

Before a ClaudeCode lane is treated as production evidence, save:

- `claude --version`;
- command path;
- exact command line;
- model and effort;
- stdout/stderr or stream log;
- output file path and write timestamp.

For thick-content production, the default is Opus with max/highest available effort. If a run uses Sonnet or an unspecified model, it is provisional and cannot become the final thick-content lane until rerun or explicitly waived by the user.

## 5. External Review Capture

For GPT Pro and Claude Opus web/app calls, save all of:

- prompt/submission packet;
- visible model/mode confirmation when available;
- URL;
- page copy or raw response;
- status note;
- Codex evidence adjudication.

Unclicked, empty, or unsaved pages do not count as received feedback. A model answer that cannot be retrieved must be marked `review_not_captured`.

## 6. Anti-Merge Rule

Core concepts may be merged for student memory. Question cases should default to one question per example.

A case may be merged only if:

- the source packet itself groups the alternatives under one scoring point; or
- every merged question has a recoverable single-question entry with all required fields.

If a merged entry cannot be traced back to independent source entries, either split from the original drafts or move it to an appendix as unresolved/combined evidence. Do not split by guessing from the final merged wording.

## 7. Final Claim Audit

Before saying "completed", "reviewed by GPT/Claude", "generated by Codex + ClaudeCode + GPT Pro", "uploaded to GitHub", or "ready for delivery", run a final claim audit.

Required checks:

- all expected artifacts exist;
- model evidence files exist for each claimed model;
- governor audit names the exact final files;
- DOCX files were rendered if delivered;
- `git status -sb` is clean or explained;
- local `HEAD`, `origin/main`, and remote branch commit match after push;
- any waived or blocked step is named, not hidden.

## 8. Xuanbier Specific Optimization

For 选必二, do not start with a final framework. Start with legal source preprocessing.

Every main-question source packet must identify:

- legal relationship;
- dispute/request;
- rule or legal element;
- fact-to-rule match;
- responsibility/procedure/effect;
- rule-of-law value if rewarded by the rubric.

Choice questions need a legal-basic-knowledge output: scenario, correct legal judgment, each wrong option's legal mistake, and the fact/rule boundary.

Framework promotion requires a proposition-path review: why the case carrier was chosen, what legal reasoning ability is rewarded, and whether the case strengthens the high-frequency trunk or exposes an open container.

## 9. Xuanbisan Specific Optimization

For 选必三, do not produce a concept glossary.

Every usable main-question row must follow:

`材料动作 -> 总帽子 -> 小方法 -> 触发逻辑 -> 答案句`

Separate:

- main-question thinking-method scoring chains;
- choice-question trap signals;
- pure formal-logic/reasoning-rule questions;
- other-book or mixed-module boundary cases.

The final student body must be framework-first. Paper order and district order are audit views, not the teaching document structure.

## 10. GitHub Sync Rule

After a substantial run or workflow-rule change:

- stage only intended files;
- commit with a clear message;
- push to the agreed sync branch;
- verify remote commit;
- include the commit id in the final handoff.

Word files may be ignored by default. If a final `.docx` is in scope, force-add it intentionally and name that in the sync report.
