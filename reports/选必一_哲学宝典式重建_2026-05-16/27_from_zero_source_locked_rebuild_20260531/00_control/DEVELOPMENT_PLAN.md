# DEVELOPMENT_PLAN

## Phase 1: Control And Inventory

- [x] 通过项目三层 SOP。
- [x] 锁定用户二级框架和哲学宝典样式源。
- [x] 建立主观题候选清单，只保留有原卷和可审核评分来源的题。

## Phase 2: Evidence Cards

- [x] 每道候选题生成证据卡：原卷设问、材料摘要、细则原文、分值、证据等级、文件路径。
- [x] 扫描 PDF 或无文本层文件必须 OCR；OCR 不清楚则记录阻塞，不进入正文。
- [x] 参考答案只能作为辅助，不能替代细则位置。

## Phase 3: Term Merge

- [x] 从证据卡抽取评分术语原词。
- [x] 按六要素和用户二级框架归类。
- [x] 同一采分核心合并，保留来源题和表述变体。

## Phase 4: Student Draft

- [x] 为每个合并核心写材料触发和答案句。
- [x] 清除后台语言：设问要求、细则要求、采分点、v7、本题需要、材料中。
- [x] 学生版正文不显示细则依据、证据层级、源文件路径；这些只进入 QA/后台证据卡。
- [x] 生成 Markdown 内容稿和 QA 表。

## Phase 5: Word

- [x] 按哲学宝典样式生成 DOCX。
- [x] 做 DOCX 结构 QA、压缩包完整性 QA、Quick Look 或 render_docx 视觉 QA。

## Acceptance

- [x] 所有入正文条目都有证据卡编号。
- [x] 所有 `术语：` 均可回指细则原文。
- [x] 后台证据卡的 `细则位置` 写清题号、分值、必答/选答/等级或证据层级；学生正文不显示该字段。
- [x] 无普通参考答案冒充评分细则。
