## 二、十五小时中暴露出的核心问题

### 1. 把材料层进度误当成论文质量进度

最早的流程在“下载到全文、能抽取文本、能建立引用映射”上投入很多，这些步骤是必要的，但它们只证明“材料可用”，不证明“论文有问题意识、论证推进和作者判断”。

这导致一个严重偏差：系统会给出很多漂亮的数字，例如全文数量、引用次数、索引覆盖、审计 PASS，但论文仍可能只是文献综述和概念拼接。

优化原则：所有材料指标只能服务于论证，不能替代论证。下一版流程必须把“研究谜题、中心判断、反解释、材料如何支持判断”放在下载和写作之间的硬闸口。

### 2. “学习优秀论文”做得太晚，而且学错对象

之前的流程把优秀论文学习放在材料收集之后、写作之前，但实际执行时更像是学习章节模板和表达套路，而不是学习论文如何制造问题意识、如何让概念咬住材料、如何处理反解释。

这也是当前初稿质量差的根源之一：形式看起来像论文，内部却缺少真正的研究动作。

优化原则：学习优秀论文必须早于正式骨架生成，但不能复制结构。只学习 craft：

- 好论文如何把大主题压成一个可研究的谜题；
- 如何让一个核心概念承担解释功能；
- 如何让材料进入比较或机制说明；
- 如何处理既有解释的不足；
- 如何在结论中保留边界，而不是泛泛喊口号。

### 3. 方法命名和材料能力不匹配

当前初稿曾经把自己写成“比较研究”，但材料并不是同一调查设计下的一组可比案例，很多材料来自既有论文、政策文本和二手案例。如果没有同维度编码表，所谓比较就只是把案例分段摆放。

优化原则：方法名必须被材料能力约束。没有原始访谈，就不能写成田野研究；没有同维度指标，就不能写成严肃比较研究；只有二手材料时，应诚实写成机制性比较、二次分析或文献-案例综合。

### 4. 外部评审 provenance 记录不够严格

用户明确要求真实调用 Claude 网页 Opus 4.8 Max 和 GPT-5.5 Pro，而不是 CLI 或预检。早期流程把 CLI 调用、登录预检、网页真实会话之间的差别记录得不够硬，容易让“看起来调用过”混同于“用户可见的真实网页评审”。

优化原则：外部评审必须区分三种状态：

- web_session：真实提交到用户可见网页会话，有 URL、截图或记录；
- cli_or_api_real_call：真实 CLI/API 调用，但不等于网页留痕；
- preflight_only：只检查登录、额度或工具状态，不能算评审。

用户要求网页评审时，只有 web_session 能通过该闸口。

### 5. 浏览器准入闸口不能靠“用户能下载”代替

用户手动登录并下载成功，只能说明账号和路径有效；它不能说明 Codex 能在不越过验证码、拼图、SSO、下载确认等限制的前提下稳定控制浏览器完成检索和下载。

优化原则：浏览器 hands-free 闸口必须独立记录：

- 是否通过人大图书馆入口进入数据库；
- 是否能读取当前页面标题和 URL；
- 是否能在授权页面内搜索；
- 是否能点击数据库内的合法下载入口；
- 遇到验证码、拼图、身份核验时是否暂停给用户处理；
- 是否没有使用知网个人登录替代人大图书馆路径。

### 6. 论文写作环节缺少“删除测试”

差论文常见问题是每段都在说“某某认为”，删掉引用句后就没有作者自己的判断。当前流程一度用引用数量、综述动词数量、段落结构来衡量质量，这些指标不足以判断论文是否真正站得住。

优化原则：正文段落必须通过删除测试：删除只报告他人观点、但不服务本文判断的句子后，段落仍应保留清楚的作者论证。

### 7. 最终闸口没有足够突出“未完成”

工作流已经能生成很多文件，很容易让目录看起来完整。但真实状态是：材料层和文本质量层可以阶段性通过，最终目标仍没通过。

优化原则：所有状态报告必须把“阶段性可用”和“最终完成”分开。不能用“paper_material_ready=yes”掩盖“final_user_goal_ready=no”。

## 三、优化后的工作流

### 第 0 步：合规和入口确认

确认数据库入口只能走用户授权的学校图书馆路径。人大场景下，优先从中国人民大学图书馆电子资源入口进入知网代理路径，不使用知网个人登录作为主路径。

遇到登录、验证码、拼图、SSO、身份核验、下载确认等环节，必须暂停，由用户完成。Codex 不解验证码，不绕过访问控制。

### 第 1 步：主题压缩和研究谜题

用户只给宽泛主题时，系统自动生成 20-30 个候选题，并评分：

- 文献可得性；
- 材料可得性；
- 研究对象清晰度；
- 方法可行性；
- 新意风险；
- 完成风险。

在正式检索前，必须写出：

- 研究对象；
- 一句话中心判断；
- 研究谜题；
- 反证条件；
- 要替换或推进的既有解释。

该步不过，不进入正式写作。

### 第 2 步：优秀论文 craft 学习

从强论文中学习写作机制，而不是照抄章节。每篇强论文只提取：

- 它如何提出问题；
- 它如何组织材料；
- 它如何处理概念；
- 它如何安排反解释；
- 它如何让作者声音压过文献堆叠；
- 它有哪些不能照搬之处。

### 第 3 步：授权检索和来源登记

生成核心概念、对象/案例、方法/理论三组检索式。每条检索记录必须保存数据库、入口路径、题名、作者、刊物、年份、摘要/关键词、可见状态和下载状态。

所有正式证据只能来自：

- 已授权下载的 PDF/CAJ/KDH 转换文件；
- 用户导出的全文；
- 数据库或期刊官方全文；
- 用户提供的材料。

题名、摘要、公共转载只能作为候选或背景，不能承载正文关键论断。

### 第 4 步：材料矩阵和方法-材料匹配

每条来源进入文献矩阵，并标记：

- title_visible；
- abstract_visible；
- full_text_read；
- PDF_or_user_exported；
- needs_user_export；
- not_usable_yet。

正式论证只使用 full_text_read 或 PDF_or_user_exported。方法命名必须由材料决定。

如果声称“比较”，必须有同维度编码表；如果没有同维度材料，只能写机制分析或二次分析。

### 第 5 步：论证骨架

正式初稿前必须形成：

- 题目；
- 研究问题；
- 中心判断；
- 分论点；
- 每节功能；
- 每节证据；
- 反解释和回应；
- 结论边界。

核心概念不超过三个。每个概念必须在正文中承担解释任务。

### 第 6 步：初稿写作

正文优先写作者判断和材料运动，引用只作为证据或对话对象。禁止把正文写成“文献接龙”。

每个机制段至少回答：

- 本段主张是什么；
- 材料如何显示这个主张；
- 为什么不是另一种解释；
- 本段如何推进全文中心判断。

### 第 7 步：本地审计

至少跑通以下闸口：

- 材料层：全文数量、正式来源、来源 ID 冻结；
- 引用层：引用编号可映射到来源；
- 页码层：正文引用有最终页码或位置锚点；
- 文本质量层：研究对象、方法-材料匹配、比较诚实、段落推进、结论边界；
- 浏览器层：人大图书馆/知网路径可控，不靠知网个人登录；
- 外部评审层：真实 Claude 和 GPT 评审。

注意：关键词页码候选只是工作锚点，不是最终页码核验。

### 第 8 步：真实外部评审

外部评审必须保存：

- 使用模型；
- 调用渠道；
- 是否网页可见；
- 会话 URL 或可核验证据；
- 原始回复；
- 结论是 pass、conditional_pass 还是 revise；
- 每条意见对应的采纳、拒绝、延后或待核验状态。

用户要求真实网页 Claude/GPT 时，CLI 或 API 不能替代。

### 第 9 步：重写和复评

如果外部评审指出结构性问题，不做补丁式修补，必须回到研究谜题、方法-材料匹配和论证骨架。

只有在以下条件同时满足时，才能宣称完成：

1. 材料层通过；
2. 文本质量层通过；
3. 页码层通过；
4. 人大图书馆/知网浏览器路径 hands-free 闸口通过；
5. Claude Opus 网页评审通过；
6. GPT-5.5 Pro 真实评审通过；
7. 总闸口矩阵显示 final_user_goal_ready=yes。

## 四、当前成果可上传范围

可以上传到 GitHub 的内容：

- 工作流说明；
- 工程化验收记录；
- 知网接入验收记录；
- Chrome 接入排错与验收记录；
- 试运行中的状态、选题、检索日志、文献矩阵、范式提取、论证骨架、论文初稿、引用审查、完成度审计、外部评审记录、总闸口矩阵、质量差距诊断和本复盘文档；
- skill 的说明文件和审计脚本；
- 安全上传范围说明。

不应上传的内容：

- CNKI/RUC 下载的 PDF、CAJ、KDH 或转换文件；
- 全文抽取文本；
- evidence_index、citation_* 等含有较多原文上下文或页码候选的文件；
- 浏览器截图、网页会话缓存、账号信息、cookie、token、日志中可能带认证痕迹的文件；
- 本地工具虚拟环境和二进制转换工具；
- 用户账号、登录路径中可能暴露隐私的原始记录。

5. 根据双评审结果继续重写，而不是只修局部句子。

## 六、底线

这个工作流已经比最初更接近可控：它能区分材料、文本、页码、浏览器、外部评审和最终验收。但它目前还不能被称为完成。下一版的核心不是继续堆更多审计文件，而是把每一步都压回一个问题：这一步是否让论文更像一篇真正有判断、有材料、有边界的研究论文。
